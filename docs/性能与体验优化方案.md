# 校园外卖小程序 - 性能与体验优化方案

> 依据 [微信小程序官方性能文档](https://developers.weixin.qq.com/miniprogram/dev/framework/) 及项目现状整理，**仅方案说明，暂不涉及改代码**。实施时可按优先级分阶段落地。

---

## 一、目标与参考指标

- **启动性能**：用户从打开小程序到首页可见的耗时。建议参考：  
  - 需下载/更新时：安卓 ≤ 3.7s，iOS ≤ 1.8s  
  - 使用本地代码包时：安卓 ≤ 2.6s，iOS ≤ 0.9s  
- **运行时性能**：列表滚动、页面切换、操作反馈流畅，无卡顿、白屏、闪退。  
- **体验**：首屏有内容（骨架屏/缓存数据）、弱网可读、符合平台体验评分要求。

---

## 二、启动性能优化

### 2.1 代码包体积（高优先级）

| 方向 | 说明 | 当前情况与建议 |
|------|------|----------------|
| **分包与主包** | 主包 ≤ 2M，首屏仅依赖主包；非首屏页面放入分包，按需下载。 | 已使用多分包（merchant / admin / order / rider / category / secondhand / store / common），结构合理。需确认主包体积（含 tab 页 home/order/cart/profile 及主包内 admin/login 等）是否接近 2M，接近则考虑将 **admin、login-example、user-info-setting** 等非首屏入口进一步迁出或从主包移除未用页面。 |
| **独立分包** | 活动页、广告页、支付结果页等可放独立分包，从该页进入不下载主包。 | 若有独立活动页/支付结果页，可单独建独立分包，减少冷启动下载量。 |
| **分包预下载** | 进入首屏后预拉下一可能访问的分包，减少切页等待。 | 已配置 `preloadRule`（home 预拉 order/store/rider，profile 预拉 common/rider 等），可结合埋点看用户真实路径，微调预拉包与时机。 |
| **分包异步化** | 主包引用分包内组件/逻辑时用异步化，避免主包膨胀。 | 若主包因引用分包组件而过大，可对部分引用改为「分包异步化」以减主包体积。 |
| **资源与无用代码** | 大图/音频/视频放 CDN；及时清理未用组件、插件、依赖、测试代码。 | 建议：① 用开发者工具「代码依赖分析」看包构成；② 代码包内仅保留小图标，大图用 URL；③ 用 `packOptions.ignore` 排除调试/文档等；④ 若有构建工具，开启 tree-shaking，避免打进未用 npm 包。 |

### 2.2 代码注入（高优先级）

| 方向 | 说明 | 当前情况与建议 |
|------|------|----------------|
| **按需注入** | 仅注入当前页面及依赖的组件/页面代码，减少注入与执行量。 | 已开启 `lazyCodeLoading: "requiredComponents"`，保持即可。 |
| **用时注入** | 在按需注入基础上，部分组件首次被渲染时才注入，进一步缩短启动与首屏。 | 可在关键首屏页面评估：将首屏不可见或低频组件改为「用时注入」（若基础库支持）。 |
| **全局组件与插件** | 在 `app.json` 的 `usingComponents` / `plugins` 中声明的会在启动时注入。 | 建议：仅全站共用的组件放 app.json；仅某分包使用的组件只在该分包/页面 json 中声明；插件尽量在分包内引用，避免主包注入。 |
| **启动阶段同步 API** | 同步 API（如 `getSystemInfoSync`、`getStorageSync`）会阻塞 JS 线程，拉长注入与首屏。 | **现状**：`app.js` 的 `checkUserLogin()` 在 onLaunch 中调用了 `getStorageSync('userInfo'/'userToken')`；`profile` 的 `buildMenuSections` 使用 `getStorageSync('adminToken'/'userToken')`；多处页面 onLoad 使用 `getStorageSync`。**建议**：① onLaunch 中仅保留最少次数的 Storage 读取（如仅读 userToken 判断登录），且可考虑异步 `wx.getStorage` 后写 `globalData`；② 对 `getSystemInfoSync`：能改用 `wx.getWindowInfo()` / `wx.getAppBaseInfo()` 等异步或拆分 API 的尽量替换，且启动路径最多调用一次并缓存结果；③ 非启动路径的页面内 getStorageSync 可保留，但避免在 App.onLaunch/首页 onLoad 之前形成长链同步调用。 |
| **启动阶段复杂运算** | 在 App/Page 定义外以及 onLaunch/onLoad/onShow 中避免重计算。 | 将复杂逻辑、大数据结构初始化放到 onReady 之后或 setTimeout 延后执行。 |

### 2.3 首屏渲染（高优先级）

| 方向 | 说明 | 当前情况与建议 |
|------|------|----------------|
| **初始渲染缓存** | 非首次启动时，视图层可用缓存先渲染，不等待逻辑层首屏数据。 | 在首页（及关键 tab 页）的 json 中配置 `"initialRenderingCache": "static"`（或按需选 dynamic），可显著缩短二次打开的首屏时间。 |
| **首屏数据精简** | data 仅放与首屏渲染相关字段；大对象、长列表不塞入 data。 | 首页 data 已分块（banners、displayProductsLeft/Right 等），可继续保证：与首屏无关的字段不放 data，用 `this.xxx` 存；首屏列表可考虑「先少后多」渐进渲染。 |
| **提前请求、缓存渲染** | 首屏依赖的接口在 onLoad 或更早发起；优先用缓存渲染，再网络更新。 | 首页已在 onLoad 调用 `loadBanners()`、`loadProducts()`，符合「提前请求」。可补充：① 对列表/轮播等优先从本地缓存取一份展示，再请求接口覆盖；② 若使用「数据预拉取/周期性更新」，可把首屏关键数据预拉，减少白屏时间。 |
| **骨架屏** | 数据未到时展示骨架结构，提升等待意愿。 | 首页有 `productListLoading`，可配合骨架屏组件（如 Vant Skeleton）在列表区展示骨架，避免长时间空白。 |
| **未使用组件** | 页面 json 中声明的组件都会参与首屏渲染，未用到的及时移除。 | 定期检查各页 `usingComponents`，删除未引用组件，减轻首屏节点数与层级。 |

## 三、运行时性能优化

### 3.1 setData 使用（高优先级）

| 方向 | 说明 | 建议 |
|------|------|------|
| **仅传变更数据** | setData 只传本次修改的路径/字段，不传整棵 data。 | 避免 `this.setData(this.data)`；列表/对象更新用路径如 `list[2].name`、`obj.a.b`。 |
| **控制频率** | 避免毫秒级、连续多次 setData；不在 onPageScroll 里每次 setData。 | 倒计时、滚动相关更新尽量收口到组件内 setData，或节流/合并后再 setData。 |
| **控制数据量** | 单次 setData 体量不宜过大，尤其避免大列表整块更新。 | 长列表采用分页、虚拟列表或「追加一段」的增量更新；与渲染无关的业务数据放 `this.xxx`。 |
| **后台页** | 页面 onHide 后尽量不 setData；若需更新可延到 onShow 后再做。 | 检查定时器、轮询、WebSocket 回调等，在 onUnload/onHide 时停止或不再 setData。 |

### 3.2 渲染与滚动（中优先级）

| 方向 | 说明 | 建议 |
|------|------|------|
| **onPageScroll** | 只要 Page 传入 onPageScroll，滚动时就会高频率发事件到逻辑层。 | 非必要不监听；需要时不用空函数占位；回调内不做重逻辑、不频繁 setData；曝光统计优先用 `IntersectionObserver` 替代「onPageScroll + SelectQuery」。 |
| **动画** | 优先 CSS 动画、WXS 响应事件；避免用连续 setData 驱动动画。 | 列表动效、滚动联动等优先 CSS/WXS；若必须 setData，尽量收口到小组件内。 |
| **节点数与层级** | 单页 WXML 节点数建议 <1000，深度 <30，单节点子节点数 ≤60。 | 长列表用虚拟列表或分屏渲染；减少无意义包裹节点，控制层级。 |
| **图片** | 控制尺寸与体积；避免 widthFix/heightFix 导致布局重排抖动。 | 列表图使用合适尺寸与格式；背景/banner 尽量预设宽高，避免加载后二次撑开。 |

### 3.3 页面切换（中优先级）

| 方向 | 说明 | 建议 |
|------|------|------|
| **onHide/onUnload** | 切换前会先执行上一页 onHide/onUnload，耗时会拖慢切换。 | 这两个生命周期内逻辑尽量轻量；耗时操作用 setTimeout 延后或放到 onShow 后。 |
| **预加载时机（安卓）** | 默认 onReady 后 200ms 预加载下个页面，可能抢占当前页渲染。 | 若在安卓上发现首屏 setData 或交互有延迟，可在 app.json 或页面 json 中配置 `handleWebviewPreload: "auto"` 或 `"manual"`，在首屏主要内容渲染完成后再预加载。 |
| **提前请求** | 从 A 跳 B 时，可在 A 里先发起 B 所需接口，通过 EventChannel 把结果传给 B。 | 对店铺详情、订单详情等重数据页，可在上一页点击时即发起请求，减少 B 页 onLoad 后的等待。 |

### 3.4 内存（中优先级）

| 方向 | 说明 | 建议 |
|------|------|------|
| **分包与按需注入** | 同启动优化，减少常驻代码与页面数，有利于内存。 | 保持现有分包与按需注入；避免主包引用过多大组件。 |
| **内存告警** | 使用 `wx.onMemoryWarning` 做降级或释放。 | 在 App 或关键页监听，释放非必要缓存、大对象、离线数据等。 |
| **泄漏** | 页面/组件销毁后仍被引用会导致无法回收。 | 检查：① 全局/模块级变量是否持有 page 的 this；② 事件监听（如 onThemeChange、自定义事件）是否在 onUnload 解绑；③ setInterval/setTimeout 是否在 onUnload 清除；④ 异步回调中是否长期持有 this。 |

## 四、体验与其它

### 4.1 体验评分与规范

- 使用开发者工具「体验评分」和「性能面板」做常规检查，按提示修复性能与体验项。  
- 遵循官方「最佳实践」：如授权文案、按钮可点区域、加载与错误态等。

### 4.2 弱网与容错

- 关键接口超时、重试、降级（如首页列表失败时展示缓存或友好提示）。  
- 图片加载失败时占位或隐藏，避免裂图影响布局。

### 4.3 版本发布节奏

- 版本更新会触发同步拉包与缓存重建，短时启动耗时会上涨。在满足需求前提下，适当控制发版频率，有利于大盘启动表现。

## 五、实施优先级建议（不改代码，仅排期参考）

1. **P0（优先）**  
   - 启动路径减少同步 Storage/SystemInfo 调用，改为异步或延后。  
   - 首页（及 tab 页）开启初始渲染缓存。  
   - 首页列表/轮播：优先缓存渲染 + 骨架屏。  
   - 检查主包体积与全局组件/插件，控制主包 ≤2M 且无多余全局依赖。

2. **P1**  
   - setData 只传变更路径、控制频率与数据量。  
   - 长列表与图片尺寸/格式、节点数控制。  
   - onHide/onUnload 精简；必要时配置 handleWebviewPreload。

3. **P2**  
   - 内存告警与泄漏排查；EventChannel 提前请求；分包预下载与异步化微调。  
   - 体验评分与弱网、发版节奏的持续优化。
