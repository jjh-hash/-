## 2026-04-04 - Web 管理端登录报 admin_sessions 集合不存在（-502005）

### 问题现象
- 静态网站 `admin-web` 能打开登录页，点击登录后报错：`collection.add:fail -502005 database collection not exists` / `Db or Table not exist: admin_sessions`。

### 根因
- 云函数 `adminManage.adminLogin` 成功后会向 **`admin_sessions`** 写入会话；该集合若从未在云开发数据库中**手动创建**，首次写入会失败（部分环境不会自动建集合）。

### 处理
- 云开发控制台 → **数据库** → **添加集合**，名称 **`admin_sessions`**；可选为字段 **`token`** 建唯一索引。

## 2026-04-06 - 楼层识别改为按门牌号首位

### 问题描述
地址表单没有单独“楼层”字段，原先通过“X楼”文本识别楼层不稳定，导致配送费和骑手收入计算不准确。

### 解决方案
1. 楼层识别改为优先读取门牌号首位（仅 3 位门牌号）：
   - `232` -> 2 楼（配送费 1.5 元，骑手收入 1.0 元）
   - `543` -> 5 楼（配送费 2.0 元，骑手收入 1.3 元）
2. 若门牌号不满足三位规则，再兼容旧逻辑从地址文本中匹配“X楼”。
3. 前端结算页与云函数统一使用同一识别规则，避免展示金额与下单金额不一致。

### 相关文件
- `subpackages/store/pages/checkout/index.js`
- `subpackages/store/pages/unified-checkout/index.js`
- `cloudfunctions/orderManage/index.js`

## 2026-04-06 - 修复地址列表点击后无法编辑

### 问题描述
在 `subpackages/common/pages/address/index` 地址列表页中，普通管理场景点击地址无响应，导致地址创建后无法再次修改。

### 解决方案
1. **地址列表点击行为分流**
   - 普通管理场景：点击地址跳转到地址编辑页（`add-address`，携带 `mode=edit` 与 `addressId`）。
   - 地址选择场景（结算/跑腿）：保持原有“回传所选地址并返回上一页”逻辑。
2. **新增地址页支持编辑模式**
   - `add-address` 页面增加 `mode/addressId` 参数读取。
   - 编辑模式下先加载当前地址详情并回填表单。
   - 点击“确定”时调用 `addressManage.updateAddress`，新增模式仍调用 `addAddress`。
3. **结算地址回传补全字段**
   - 选择地址回传时同步写入 `addressDetail/buildingName/houseNumber`，避免后续楼层配送费计算缺字段。

## 2026-04-03 - Web 管理端（admin-web）对齐小程序管理端能力

### 背景与目标
- 在仓库 `admin-web/`（Vue3 + Vite，`base: '/dist/'`）内扩展与小程序分包管理端一致的菜单与业务能力，请求体与小程序相同：`action` / `data`，并由 `src/cloud.js` 的 `callAdminFunction` 自动附带 `adminSessionToken`。

### 实现摘要
- **布局与路由**：`layouts/AdminLayout.vue` 侧栏导航 + 嵌套路由；登录守卫沿用 `getStoredSessionToken()`。
- **接口封装**：`src/api/admin.js` 按云函数名与 action 封装，便于与 `subpackages/admin/pages/*` 对照。
- **页面**：概况、商家/订单/用户列表与详情、退款详情、骑手审核、财务与多 Tab 统计、系统设置（平台配置/邀请码/轮播/公告列表）、轮播编辑（含上传）、独立公告管理、批量审核、操作日志、修改密码。
- **云存储**：`src/cloud.js` 增加 `uploadBannerImageFile` / `getTempFileUrls`，浏览器上传后拿到 fileID 再调 `bannerManage`（与小程序 `wx.cloud.uploadFile` 路径 `banners/` 对齐）。
- **登录用户名**：成功登录后 `localStorage` 保存 `adminUsername`，修改密码时传给 `adminManage.changePassword`（与小程序逻辑一致）。

### 小程序分包减负说明
- **当前未移除** `app.json` 中 `admin` 分包：个人中心、商家注册等仍跳转分包内管理页；若后续仅使用 Web 管理端，需同步修改这些入口（如改为打开 H5 静态站）后再删除分包，避免链接失效。

### 已知差异
- 小程序部分页面调用 `adminLogManage` 的 `action: 'record'`，云函数实际仅实现 **`create`**；Web 端未重复打客户端日志，避免无效请求。

## 2026-04-03 - 管理端 Web 分离、admin_sessions 与会话鉴权

### 背景

## 2026-03-10 - 重新申请校园兼职点「确认缴纳」报 E11000 duplicate key (deposit_orders openid)

### 问题现象
- 用户重新申请校园兼职后，在「待缴纳保证金」页点击「确认缴纳」时报错：`E11000 duplicate key error collection: ... deposit_orders index: openid dup key: { ... }`。

### 根因
- 集合 `deposit_orders` 对字段 `openid` 建了**唯一索引**，同一 openid 只能有一条记录。
- 每次点击「确认缴纳」时，云函数 `createDepositPayment` 都会执行 `deposit_orders.add` 插入新记录；用户首次已产生过一条（无论是否支付成功），再次申请时再插入即触发重复键。

### 处理
- **cloudfunctions/campusPartnerManage/index.js**：在 `createDepositPayment` 中，先按 `openid` + `status: 'pending'` 查询是否已有待支付单；若有则**复用**该条（用其 _id、outTradeNo 重新拉取支付参数），不再插入；若无再 `add` 新记录。这样同一用户未完成的缴纳不会重复插入。
- **数据库**：`deposit_orders` **不要对 openid 建唯一索引**，只对 `outTradeNo` 建唯一即可；同一用户可有多条记录（首次缴纳、退款后再次缴纳等）。若已建 openid 唯一索引，需在云开发控制台 → 数据库 → deposit_orders → 索引管理 中删除该唯一索引。
- **数据库集合文档.md**：索引建议改为仅 `outTradeNo` 唯一，并注明不要对 openid 建唯一索引。

### 涉及文件
- `cloudfunctions/campusPartnerManage/index.js`
- `数据库集合文档.md`

## 2026-03-10 - 申请退还保证金只显示「ok」且未实际退款

### 问题现象
- 在校园兼职页点击「申请退还保证金」后，只弹出「ok」提示，款项未原路退回。

### 可能原因
1. **前端拿到的返回值是 getStatus 的 200 + message: 'ok'**（例如云函数未按 requestRefund 执行或返回被包装），导致提示为「ok」且未走退款逻辑。
2. **campus_partners 记录缺少 depositOrderId**：只有通过「确认缴纳」流程（支付成功后调用 confirmDeposit）才会写入 depositOrderId；若记录是旧数据或未走该流程，会走「无支付记录」分支，只解除身份、不调用微信退款，此时应显示「已解除校园兼职身份（该笔保证金无支付记录，未执行原路退）」而非「ok」。
3. **paymentManage 退款失败**（证书未配置、商户号/订单号异常等）：云函数会返回 500 和错误信息，前端应显示该错误信息。

### 处理
- **前端 subpackages/common/pages/campus-partner/index.js**：对云函数返回做兼容；若 message 为 'ok' 则改为提示「已提交，请查看页面状态」；并 `console.log('申请退还保证金 云函数返回:', result)` 便于排查。
- **云函数 campusPartnerManage**：解析 paymentManage 返回值时兼容 `refundRes.result` 与直接 `refundRes`，避免云函数互调时结构差异导致误判。

### 排查建议
- 在开发者工具控制台查看「申请退还保证金 云函数返回」的完整 result（code、message、data）。
- 在云开发控制台查看 campusPartnerManage 请求日志，确认 action 是否为 requestRefund 以及返回值。
- 在数据库中确认该用户的 campus_partners 是否有 depositOrderId，deposit_orders 对应单是否 status=paid 且含 outTradeNo、amountFen。
- 若需原路退款，确认 paymentManage 已配置微信支付商户证书（refundByOutTradeNo 需要）。

### 补充（code 500 + message "OK"）
- 微信支付退款 API 在业务失败时常用 return_msg: "OK"（表示请求已收到），真实原因在 err_code_des。若 paymentManage 用 return_msg 作为错误文案，会得到 message: "OK"，用户看到「ok」且未退款。
- **处理**：paymentManage 退款失败时优先使用 err_code_des 作为 message，并忽略无意义的 "OK"；campusPartnerManage 收到 500 且 message 为 "OK" 时改为提示「退款失败，请确认商户证书与退款配置或稍后重试」。paymentManage 中增加 console.warn 输出微信返回的 err_code、err_code_des 便于在云函数日志中排查。

## 2026-03-10 - 管理端配送费设置不生效（展示与实付与平台配置不一致）

### 问题现象
- 在管理端「系统设置」页面（subpackages/admin/pages/admin-settings）修改「配送费」后，前端展示与订单支付仍按 3 元，未按管理端设置收款。

### 根因
- 管理端修改的配送费写入 **platform_config** 集合（单位：分），**orderManage** 创建订单时已从 platform_config 读取配送费，实付金额本身是按平台配置计算的。
- 前端展示的配送费来自 **getStoreList**、**storeManage**（getStoreDetail、getStoreDetailWithProducts、getStoreInfo），这些接口返回的 `deliveryFee` 取自 **stores** 表或写死默认值 3，未使用 platform_config，导致列表/店铺详情/结算页显示与实付不一致，用户误以为「设置没用」。

### 处理
- **cloudfunctions/getStoreList/index.js**：在返回店铺列表前查询 platform_config，取 `deliveryFee`（分）转为元，列表项中 `deliveryFee`、`delivery` 使用该平台配送费（元），不再使用 `store.deliveryFee || 3`。
- **cloudfunctions/storeManage/index.js**：
  - 新增辅助函数 `getPlatformDeliveryFeeYuan()`，从 platform_config 读取配送费（分）并返回元。
  - **getStoreDetail**、**getStoreDetailWithProducts**、**getStoreInfo** 返回的店铺信息中 `deliveryFee` 改为使用平台配送费（元），与管理端设置一致。

### 涉及文件
- `cloudfunctions/getStoreList/index.js`
- `cloudfunctions/storeManage/index.js`

### 说明
- 订单创建（orderManage）已使用 platform_config，无需修改。修复后首页、店铺详情、结算页展示的配送费与订单实付均与「系统设置」中的配送费一致。

## 2026-02-27 - 商户端订单与提现统计核对（统一付款不漏单）

### 核对结论
商户端订单列表、提现可提现余额、账户余额、销售统计**全部基于 orders 集合**，按 **storeId**（当前商户店铺）和 **payStatus**（如 `paid`）、**orderStatus**（如排除 `cancelled`）查询，**不依赖** `unified_payments` 或 `unifiedPaymentId`。  
统一付款产生的多笔订单在 `orders` 中各自存在且支付成功后均被置为 `payStatus: 'paid'`，因此**不会被漏掉**。

### 核对过的位置
| 功能 | 位置 | 查询方式 |
|------|------|----------|
| 商户订单列表 | `orderManage.getMerchantOrders` | `orders.where({ storeId, payStatus: 'paid' })`（前端传 payStatus） |
| 可提现余额 | `withdrawManage.computeAvailableBalance` | `orders.where({ storeId, payStatus: 'paid', orderStatus: neq('cancelled') })`，用 amountGoods 汇总 |
| 账户余额/总收益 | `statistics.getAccountBalance` | 同上，总收益再扣退款、已提现 |
| 销售概览/图表/趋势 | `statistics.getSalesOverview` 等 | `orders.where({ storeId: targetStoreId, ... })`，无 unifiedPaymentId 过滤 |

### 结论
无需修改代码；统一付款订单在商户端展示与提现统计中均会正确计入。

## 2026-02-27 - 取消订单云函数超时（-504003）

### 现象

2. 左侧 **云函数** → 找到 **orderManage** → 点击进入；
3. 在「配置」或「函数配置」里找到 **超时时间**，由默认 3 秒改为 **20** 秒（或 60 秒），保存。

若使用腾讯云控制台：云开发 → 云函数 → orderManage → 编辑 → 高级配置 → 执行超时时间 → 改为 20 秒。

### 说明
paymentManage（退款）也建议将超时时间调大（如 20 秒），因调微信退款 API 可能较慢。

## 2026-02-23 - 商家端添加商品保存报 403「商家不存在」

### 问题现象
- 商家端「添加商品」保存时云函数返回 403，提示「商家不存在」；前端已正确从 storage 取 merchantInfo 并传 merchantId。

### 原因
- 云函数在通过传入的 merchantId 查到商家文档后，仍要求 `merchant.openid === 当前请求的 OPENID`；在模拟器、多环境或 openid 不一致时，会误判为「商家不存在」并返回 403。

- **productManage**（addProduct）：商家校验改为「有 clientMerchantId 且按 id 查到商家即通过」，不再用 openid 二次校验。merchantId 来自商家登录后写入的 storage，仅登录商家端可拿到，在此前提下信任传入的 merchantId。
- 前端（主包/分包添加商品页）：保存时传 `merchantId: merchantInfo._id`，已具备；无需改业务逻辑。

### 涉及文件
- `cloudfunctions/productManage/index.js`（addProduct 内商家校验逻辑）

### 建议
- 若后续需区分「仅允许操作自己账号下数据」，可在业务层用 openid 做二次校验并给出明确提示，而不是在「商家是否存在」这一步用 openid 拦截。

## 2026-02-23 - 首页报错 @babel/runtime/helpers/defineProperty.js is not defined

### 问题现象
- 打开首页时报错：`Error: module '@babel/runtime/helpers/defineProperty.js' is not defined, require args is './defineProperty'`
- 控制台提示页面 `pages/home/index` 错误，伴随 `Component is not found in path "wx://not-found"`。

### 原因
- 微信开发者工具在编译时会做 ES6→ES5 转译。页面中使用了**对象展开**（`...obj`）、**空值合并**（`??`）等语法时，会注入对 `@babel/runtime/helpers/defineProperty.js` 等模块的 `require`。
- 小程序运行环境下对 npm 包路径的解析与普通 Node 不同，导致该 require 解析失败，从而报「模块未定义」。

### 处理
- 在 **pages/home/index.js** 中避免会触发 Babel 注入 runtime 的语法，改为不依赖 @babel/runtime 的等价写法：
  - `a ?? b` → `a != null ? a : b`
  - `return { ...obj, key: val }` → `return Object.assign({}, obj, { key: val })`
  - `[...arr1, ...arr2]` → `arr1.concat(arr2)` 或 `arr.slice()`
  - `Math.max(...arr)` → `Math.max.apply(Math, arr)`
  - `setData({ ...payload, other: x })` → 先 `Object.assign` 合并对象再 `setData`
- 修改后首页不再依赖 `@babel/runtime` 的 defineProperty/objectSpread，首页可正常加载。

### 涉及文件
- `pages/home/index.js`

### 建议
- 若其他页面也出现同类 `@babel/runtime/helpers/xxx is not defined` 报错，可按同样方式改写该页面的对象展开、空值合并等语法，或确保在微信开发者工具中已正确「构建 npm」且 miniprogram_npm 中包含 @babel/runtime。

## 2026-02-22 - 页面跳转卡顿与「滑动式」流畅实现说明

### 问题现象
- 页面跳转感觉卡顿，不跟手；其他项目采用「滑动」效果更流畅。

### 原因与结论
1. **主要原因：Tab 切换用了 reLaunch**  
   首页 / 订单 / 我的等底部 Tab 通过 `wx.reLaunch` 切换。reLaunch 会关闭所有页面再打开目标页，**没有推入推出动画**，视觉上像整页替换，所以感觉卡。
2. **其他因素**：跳转到分包页面时若分包未下载会有等待；当前未启用 Skyline，使用的是默认 WebView 转场。
3. **别人「滑动流畅」的常见做法**：使用 **tabBar + wx.switchTab**（系统自带过渡），或单页内 **swiper 滑动**不做真跳转，或 **Skyline 自定义路由** 优化 navigateTo 的动画。

- 新增 **《页面跳转与流畅滑动说明.md》**：说明卡顿原因、三种实现方式（switchTab / swiper / Skyline 自定义路由）、按优先级的优化建议。
- 新增 **utils/skylineRoute.js**：可选使用，在启用 Skyline 后为 navigateTo 提供「从右向左滑入」的自定义转场；未启用时自动退化为普通 navigateTo。

### 建议优先做
- **减少 reLaunch**：将主 Tab（首页、订单、我的）改为 app.json 的 tabBar 页面，用 **wx.switchTab** 切换，体感提升最明显。

外卖订单原逻辑将商品金额与配送费一起走线上支付，且骑手收入统计使用固定值，导致与当前业务规则不一致：
- 用户应只在线支付配送费；
- 餐费由骑手垫付后线下向用户收取；
- 骑手收入需按楼层计算而非固定金额。

### 解决方案
1. **统一金额口径（云函数）**
   - `createOrder`、`createUnifiedOrders` 改为仅收配送费（`amountPayable = amountDelivery`）。
   - 保留 `amountGoods` 仅用于展示代买金额，不参与在线支付。
   - 新增订单字段：`deliveryFloor`（识别楼层）、`riderIncome`（骑手结算收入，分）。

2. **楼层配送费规则**
   - 1~3 楼：配送费 1.5 元（150 分）
   - 4~6 楼：配送费 2.0 元（200 分）
   - 无法识别楼层时按 2.0 元处理。

3. **骑手收入规则**
   - 1~3 楼：1.0 元/单
   - 4~6 楼：1.3 元/单
   - 骑手今日/周/月/总统计与收入明细改为基于订单 `riderIncome` 汇总，不再使用固定 1 元/2 元。

4. **前端结算页同步**
   - `checkout` 与 `unified-checkout` 按地址楼层重算配送费。
   - 支付金额以云函数返回的 `amountPayableFen` 为准，避免前后端口径不一致。
