# 校园外卖 - 微信小程序性能优化计划

> 从高级性能优化工程师视角，针对当前项目制定的性能优化方案。  
> 编制日期：2026-02-17

---

## 一、现状与目标

### 1.1 项目特征（基于代码扫描）

- **包体与分包**：主包含首页、个人中心、管理端壳等；业务分布在 merchant / order / store / rider / admin / category / common / secondhand 等分包，已开启 `lazyCodeLoading: "requiredComponents"`，有利于首包体积控制。
- **数据与网络**：大量使用 `wx.cloud.callFunction`（订单、店铺、评价、用户等）；订单页存在 **实时 Watch + 轮询兜底**，存在频繁回调与潜在 setData。
- **渲染**：列表页普遍使用 `scroll-view` + `wx:for` 渲染订单/店铺/评价等，无虚拟列表或分页截断；部分页面一次性 setData 整表（如评价列表、财务图表数据）。
- **日志**：页面与云函数内存在大量 `console.log/warn/error`，上线后仍会执行，占用主线程与日志缓冲。
- **图片**：首页/店铺等使用网络图（含 picsum 占位）、云开发 fileID，未统一懒加载或尺寸控制。
- **生命周期**：部分页面 onShow 重复请求（已有防抖的如首页 1 分钟刷新），部分页面 onLoad/onShow 双拉取。

### 1.2 优化目标

- **首屏与包体**：主包 & 首屏分包体积可控，首屏可交互时间（TTI）缩短。
- **列表与交互**：长列表不卡顿、滚动流畅，setData 单次数据量与时序可控。
- **网络与云**：云函数调用次数与耗时可控，避免重复请求与无效轮询。
- **运行态**：减少主线程占用（日志、大对象、频繁定时器），避免内存与监听泄漏。

## 二、优化维度与建议

### 2.1 包体与首屏（Bundle & First Screen）

| 项 | 现状/风险 | 建议 | 优先级 |
|----|-----------|------|--------|
| 主包体积 | 主包含多页与可能未用页面 | 确保未用页面从主包移除或改为分包；主包仅保留首页、个人中心、登录占位、管理端 tab 壳等必要路径 | 高 |
| 分包预下载 | 未配置 preloadRule | 在 app.json 配置 `preloadRule`：从首页预下载 order、store 分包；从个人中心预下载 common；从任务大厅预下载 order | 中 |
| 首屏请求 | 首页 onLoad 同时拉 banner + 店铺列表 | 优先拉店铺列表（首屏主体），banner 可稍后或与列表并行但控制并发数；考虑骨架屏占位 | 中 |
| 静态资源 | 图片/字体未压缩或外链过大 | 图片放 CDN 或云存储并带尺寸/质量参数；避免主包内大图；图标优先 iconfont 或 SVG | 中 |

### 2.2 setData 与渲染（核心性能）

| 项 | 现状/风险 | 建议 | 优先级 |
|----|-----------|------|--------|
| setData 体量 | 订单/评价/店铺列表整表 setData | 只 setData 渲染所需字段；大列表分页加载，每页 setData 仅追加或替换当前页；避免把整份 `orders`/`reviews` 全量更新 | 高 |
| setData 频率 | 订单页 Watch 回调、轮询、滚动可能触发多次 setData | Watch/轮询合并为「有变更再刷新」，且刷新时只 setData 列表及相关状态；防抖/节流（如 300ms）再 setData | 高 |
| 长列表 | 订单/任务大厅/店铺/评价等 wx:for 全量渲染 | 列表采用分页加载（上拉加载更多）；超过一屏的列表考虑虚拟列表或只渲染可见区（如 recycle-view 或官方推荐方式） | 高 |
| 列表 key | 部分列表 wx:key 使用 index 或可能重复 id | 统一使用稳定唯一 id 作为 wx:key，避免重排与闪烁 | 中 |

### 2.3 网络与云函数（Network & Cloud）

| 项 | 现状/风险 | 建议 | 优先级 |
|----|-----------|------|--------|
| 重复请求 | onLoad + onShow 双拉、Tab 切换重复拉 | 请求前检查「是否已有数据 + 时间戳/版本」再决定是否请求；列表页用「页码 + 是否有更多」控制，避免重复拉第一页 | 高 |
| 云函数聚合 | 首屏或单页多次 callFunction（如 loginUser + getStoreList + getBanner） | 能合并的用云函数聚合接口（如首页数据一次返回 banner+stores）；减少 roundtrip 次数 | 中 |
| 订单 Watch | 实时 Watch 常驻 + 轮询兜底 | 保持 Watch 为主、轮询为辅；页面 onHide/onUnload 必须停止 Watch 与轮询；轮询间隔不宜过短（建议 ≥5s） | 高 |
| 云函数内逻辑 | 云函数内存在大量 console、未做索引查询 | 云函数日志生产环境可关闭或降级；数据库查询必须走索引，避免全表扫描 | 中 |

### 2.4 图片与资源（Image & Assets）

| 项 | 现状/风险 | 建议 | 优先级 |
|----|-----------|------|--------|
| 图片懒加载 | 未统一使用 lazy-load | 列表内图片使用 `loading="lazy"` 或微信 `lazy-load` 属性；首屏外图片延迟加载 | 高 |
| 图片尺寸 | 大图原图展示 | 云存储 URL 带尺寸/裁剪参数；列表缩略图使用小图，详情再考虑大图 | 中 |
| 占位与失败 | 部分图片无占位或失败图 | 统一占位图与 binderror 降级，避免裂图或空白 | 低 |

### 2.5 运行态与内存（Runtime & Memory）

| 项 | 现状/风险 | 建议 | 优先级 |
|----|-----------|------|--------|
| console | 页面与云函数大量 console | 生产环境通过构建或全局封装去掉/降级 console；或使用条件编译仅在开发环境输出 | 高 |
| 定时器与 Watch | 订单页多组 setTimeout/watch 需在 onUnload 清理 | 已在部分页面清理；全项目排查 onHide/onUnload 中 clearTimeout、stopWatch、stopPolling 是否成对出现，避免泄漏 | 高 |
| 全局数据 | 大对象放 globalData 或持久化 | 避免在 app.globalData 存大列表；大对象考虑按需加载或只存 id 等引用 | 中 |

### 2.6 体验与感知（Perceived Performance）

| 项 | 建议 | 优先级 |
|----|------|--------|
| 骨架屏 | 首屏列表/订单/店铺用骨架屏占位，减少白屏感 | 中 |
| 加载态 | 列表/按钮加载态统一且不阻塞（如列表底部 loading，不整页遮罩） | 中 |
| 错误与重试 | 网络失败给出明确提示与「重试」按钮，避免用户多次刷新 | 低 |

## 三、分阶段实施建议

### Phase 1：高收益、低成本（优先落地）

1. **setData 瘦身**  
   - 订单列表、任务大厅、评价列表、店铺列表：只 setData 当前页或增量数据；单次 setData 的 data 大小建议 < 256KB。  
   - 合并同一事件循环内的多次 setData 为一次（如 Watch 回调内先合并变更再 setData）。

2. **请求防重与顺序**  
   - 所有列表/详情页：在 onLoad/onShow 里加「请求锁 + 时间戳/版本」判断，避免重复请求。  
   - 首页：优先拉店铺列表，banner 可略滞后或与列表并行（控制并发）。

3. **订单页 Watch/轮询与清理**  
   - 确认 onHide/onUnload 中正确停止 Watch 与轮询、清理所有相关定时器。  
   - 轮询间隔调大到 ≥5s，Watch 连接失败后的重连间隔也适当放大。

4. **生产环境去掉 console**  
   - 通过工程化（如 minifier 配置、或封装 `console.log` 为开发环境才执行）去掉或关闭生产环境 console。

### Phase 2：列表与包体

1. **长列表分页与虚拟列表**  
   - 订单、任务大厅、店铺、评价等：严格分页（上拉加载更多），禁止一次拉全量再 setData。  
   - 超过一屏的纯列表考虑虚拟列表（需评估引入成本与兼容性）。

2. **分包预下载**  
   - 在 app.json 配置 preloadRule（如从首页预下载 order、store）。

3. **图片懒加载与尺寸**  
   - 列表内图片统一 lazy-load；云存储 URL 使用尺寸参数。

### Phase 3：云与架构

1. **云函数聚合与索引**  
   - 首页等可考虑「首页数据」聚合接口；所有数据库查询确认索引覆盖。

2. **骨架屏与加载态**  
   - 首屏与主要列表页增加骨架屏；统一列表底部加载与错误态。

3. **全局数据与内存**  
   - 排查 globalData 与持久化内容，避免存放大列表或大对象。

## 四、验收与监控建议

- **性能面板**：使用微信开发者工具「性能」与「体验」评分，关注 setData 次数与数据大小、首屏时间、包体积。
- **真机**：在低端机（如旧款 Android）上测试首屏、列表滚动、订单页切换，观察是否卡顿或白屏。
- **指标建议**：首屏可交互 < 2s、setData 单次 < 256KB、主包 < 2MB、分包按需且总包在平台限制内。

---

## 五、小结与优先级表

| 优先级 | 方向 | 具体动作 |
|--------|------|----------|
| 高 | setData | 列表只 setData 当前页/增量；合并短时间内的 setData |
| 高 | 请求 | 防重（请求锁+时间戳）；订单页 Watch/轮询与 onUnload 清理 |
| 高 | 生产环境 | 去掉或关闭生产环境 console |
| 中 | 列表 | 分页加载、图片懒加载、分包预下载 |
| 中 | 云 | 云函数聚合、索引检查、轮询间隔调大 |
| 低 | 体验 | 骨架屏、错误重试、图片尺寸/占位 |

实施时建议按 Phase 1 → Phase 2 → Phase 3 推进，每阶段在「问题修改总结文档」中记录改动与验证结果，便于回溯与持续优化。
