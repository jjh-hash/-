# 页面跳转卡顿原因与「滑动式」流畅实现说明

## 一、为什么我们的页面跳转会卡顿？

### 1. 大量使用 `wx.reLaunch`（最主要原因）

项目中底部 Tab（首页、订单、我的等）的切换是通过 **reLaunch** 实现的，例如：

- 首页 → 点「订单」→ `wx.reLaunch({ url: '/subpackages/order/pages/order/index' })`
- 首页 → 点「我的」→ `wx.reLaunch({ url: '/pages/profile/index' })`

**reLaunch 的特点：**

- 会**关闭当前所有页面**，再打开目标页面
- **没有页面推入/推出的滑动动画**，视觉上像是「闪一下」或「整页替换」
- 每次都会重新加载目标页面，无法保留上一页状态
- 所以会明显感觉「卡」「不跟手」，不像滑动切换

别人家「滑动很流畅」的，多半是：

- 要么用了**真正的 tabBar + switchTab**（系统自带过渡），  
- 要么是**一个页面里用 swiper 左右滑**，根本没有做页面跳转。

### 2. 跳转到分包页面时的「等待」

使用 `wx.navigateTo` 跳转到**分包页面**（如店铺详情、订单列表）时：

- 如果该分包**尚未下载**，会先**下载分包**再打开页面
- 这段时间用户会看到白屏或等待，感觉像「卡了一下」

你们已经在 `app.json` 里配置了 `preloadRule`，会预加载 order、store 等分包，能缓解但无法完全消除首次进入的等待。

### 3. 默认 WebView 转场动画

当前项目：

- 未启用 **Skyline** 渲染（`project.private.config.json` 中 `skylineRenderEnable: false`）
- 使用的是传统 **WebView** 下的默认页面转场
- 默认只有「从右向左推入」一种动画，且动画在 JS 线程参与较多，容易受逻辑影响，显得不够顺滑

### 4. 小结：卡顿来源

| 原因           | 表现           | 对应方案（见下文）     |
|----------------|----------------|------------------------|
| Tab 用 reLaunch | 无滑动、整页替换 | 改用 switchTab 或单页 swiper |
| 分包首次加载   | 进入分包页时短暂白屏/卡顿 | 预加载 + 骨架屏/loading |
| 默认 WebView 转场 | 动画单一、不够顺滑 | 可选：Skyline + 自定义路由 |

## 二、「滑动那种效果」一般是怎么实现的？

### 方式一：真正的 TabBar + switchTab（推荐优先考虑）

- 把「首页」「订单」「我的」等放进 `app.json` 的 **tabBar.list**
- 切换时用 **wx.switchTab** 而不是 reLaunch
- **特点：** 系统自带 tab 切换动画，无需自己写转场；切换时不会重建整个页面栈，更流畅

你们当前是**自定义底部栏** + reLaunch 模拟 tab，所以没有系统自带的滑动感。若业务允许，尽量让主入口（首页/订单/我的）作为 tabBar 页面，用 switchTab 跳转。

### 方式二：单页内用 swiper 做「伪跳转」（无页面栈）

- 不真的跳转页面，在一个页面里用 **swiper** 或 **scroll-view** 左右滑动切换「首页 / 订单 / 我的」等视图
- **特点：** 没有页面栈切换、没有分包加载等待，所以会非常顺滑；但需要把多页逻辑收进一个页面，结构会变

很多「滑动特别顺」的小程序，在主要 Tab 区域用的就是这种思路。

### 方式三：Skyline 自定义路由（仅影响 navigateTo 的推入/推出动画）

- 使用 **Skyline** 渲染 + **自定义路由**（`wx.router.addRouteBuilder`）
- 在 **navigateTo** 时传入 **routeType**，用 worklet 在 UI 线程做「从右向左滑入」等动画
- **特点：** 只优化「页面推入/推出」的动画流畅度，**对 reLaunch 无效**（reLaunch 本身就没有过渡动画）

适合：保留现有页面结构，只希望「点进详情、返回」这类 navigateTo 的滑动更跟手、更顺。

## 三、针对本项目的优化建议（按优先级）

### 1. 减少 reLaunch，能改用 switchTab 的尽量改（效果最明显）

- **首页、订单、我的** 如果是主要 Tab，建议放进 **tabBar**，用 **wx.switchTab** 跳转。
- 只有「彻底离开当前业务线」（如从用户端跳到商家端、或重新登录）时再用 reLaunch。

这样 Tab 之间就会有系统自带的过渡，而不是整页替换，体感会顺很多。

### 2. 保持并合理使用预加载

- 已有 `preloadRule`：首页预加载 order、store；个人页预加载 common；订单页预加载 store。
- 可根据用户行为再考虑是否预加载 category、secondhand 等，减轻首次进入分包时的卡顿。

### 3. 可选：启用 Skyline + 自定义路由（只优化 navigateTo 的滑动）

- 在 **project.private.config.json** 中把 **skylineRenderEnable** 设为 **true**（或按需在部分页面开启 Skyline）。
- 在 app 里注册一个**自定义路由 builder**（例如「从右向左滑入」），在需要的地方用 **navigateTo + routeType** 跳转。
- 这样**从列表进详情、详情返回**这类路径的滑动动画会更流畅，但**不会改变 reLaunch 的体验**。

我们已在 `utils/skylineRoute.js` 中提供了一个最小可用的自定义路由示例。

**接入步骤：**

1. **启用 Skyline**  
   - 在 `project.private.config.json` 的 `setting` 中设置 `"skylineRenderEnable": true`，或  
   - 仅在部分页面使用：在该页面的 json 中增加 `"renderer": "skyline"`（需基础库支持）。

2. **在 App 中注册路由（可选）**  
   在 `app.js` 的 `onLaunch` 里调用一次即可：
   ```js
   const skylineRoute = require('./utils/skylineRoute.js');
   skylineRoute.registerSkylineRoute();
   ```

3. **需要滑动转场时用封装方法跳转**  
   将原来的 `wx.navigateTo` 改为：
   ```js
   const skylineRoute = require('../../utils/skylineRoute.js'); // 路径按实际调整
   skylineRoute.navigateToWithSlide({ url: '/subpackages/store/pages/store-detail/index?storeId=' + id });
   ```
   未启用 Skyline 时会自动退化为普通 `wx.navigateTo`，不影响功能。

**注意：** 自定义路由只对 **navigateTo** 生效；**reLaunch / switchTab** 的体验需通过「减少 reLaunch、改用 switchTab」来优化。

## 五、已实施的滑动转场（Skyline）

项目已开启 **Skyline 渲染** 并接入 **自定义路由**，实现「从右向左滑入」的页面转场：

1. **开启方式**  
   - `project.private.config.json`：`skylineRenderEnable: true`  
   - `app.json`：`"renderer": "skyline"`、`rendererOptions.skyline.defaultDisplayBlock: true`

2. **使用方式**  
   - 在 `app.js` 的 `onLaunch` 中注册自定义路由，并挂载 `getApp().navigateToWithSlide(options)`。  
   - 需要滑动效果的页面跳转统一使用：`getApp().navigateToWithSlide({ url: '...' })`，已用于：首页→店铺详情/分类、订单列表→订单详情、店铺列表→店铺详情、店铺详情→结算/评价、订单详情→退款/评价/结算等。

3. **验证**  
   - 开发者工具：详情 → 本地设置 → 勾选「开启 Skyline 渲染调试」，运行后左上角会显示「当前渲染模式：Skyline」。  
   - 基础库建议 ≥ 2.31.0；真机需微信客户端 ≥ 8.0.40。

### 4. 不推荐：为「像 App 一样滑动」而全改成单页 swiper

- 除非产品决心把主流程收进一个页面，否则不建议仅为了「滑动感」大改架构；优先用 **switchTab + 少用 reLaunch** 就能明显改善。

---

## 四、总结

- **卡顿的主要原因：** Tab 用 reLaunch（无过渡动画）+ 分包首次加载 + 默认 WebView 转场。
- **别人「滑动很流畅」的常见实现：**  
  - 要么 **tabBar + switchTab**，  
  - 要么 **单页 swiper 滑动**，  
  - 再配合 **Skyline 自定义路由** 优化普通页面的推入/推出。
- **建议优先做：** 把主 Tab 改为 tabBar + switchTab，减少 reLaunch；再按需开启 Skyline 与自定义路由，提升 navigateTo 的滑动观感。
