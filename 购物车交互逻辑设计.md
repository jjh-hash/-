# 购物车交互逻辑设计

> 以「高级交互逻辑设计师」视角，在现有店铺详情 + 结算流程基础上，设计全局购物车与 Tab 购物车页的完整交互逻辑。  
> 关联：`subpackages/store/pages/store-detail`、`subpackages/store/pages/checkout`、主包 `pages/cart/index`。

---

## 一、设计目标与原则

### 1.1 目标

- **统一入口**：用户可在「购物车」Tab 看到所有已加购商品（按店铺分组），再选择某店去结算，无需每次在店铺内才能结算。
- **跨次留存**：加购数据持久化（本地或云），关闭小程序或切换店铺后再打开，购物车仍存在。
- **兼容现有**：保留「在店铺详情内加购 → 底部点结算 → 直接进结算页」的短路径；该路径与「从购物车页选店去结算」共用同一套结算页与订单逻辑。
- **单店一单**：校园外卖一单只对应一个店铺，结算页只接受单店 `cartData`；购物车可存多店，但每次「去结算」只带当前所选店铺的数据。

### 1.2 原则

- **不破坏已有流程**：店铺详情内的加购、改数量、弹层内结算按钮行为保持不变，仅增加「同步到全局购物车」与「从购物车页发起结算」。
- **数据单一来源**：全局购物车为权威数据源；店铺详情页进入时从全局购物车「拉取该店数据」以展示，离开或加购/删减时「回写」该店数据到全局购物车。
- **反馈明确**：加购成功、删除、未达起送、店铺休息、商品失效等均有清晰提示与可操作引导。

## 二、数据模型

### 2.1 全局购物车（本地存储建议）

**存储键**：`globalCart`（或 `cart`）  
**存储方式**：`wx.setStorageSync('globalCart', data)` / `wx.getStorageSync('globalCart')`  
**结构**：按店铺聚合，每个店铺一个「桶」。

```json
{
  "storeId_xxx": {
    "storeId": "xxx",
    "storeInfo": {
      "_id": "xxx",
      "storeId": "xxx",
      "name": "店铺名称",
      "deliveryFee": 2,
      "minOrder": 12,
      "businessStatus": "open",
      "avatar": "...",
      "logo": "..."
    },
    "deliveryType": "delivery",
    "items": [
      {
        "id": "商品ID",
        "name": "商品名",
        "price": 17,
        "image": "...",
        "quantity": 2,
        "selectedSpec": { "spiciness": "微辣", "portion": "小份" },
        "selectedSpecs": { "0": 1, "1": 0 },
        "finalPrice": "18.00"
      }
    ],
    "cartTotal": 36,
    "updatedAt": 1234567890
  }
}
```

- **storeId**：与 `storeInfo._id` / `storeInfo.storeId` 一致，作为 key 和校验用。
- **storeInfo**：进入店铺详情加购时写入（首次加购该店时从当前页 `storeInfo` 拷贝必要字段）；购物车页、结算页仅读。
- **deliveryType**：外送/自取，与店铺详情内选择一致；在店铺详情变更时同步回写。
- **items**：当前店铺在购物车中的商品列表；每项需包含结算页与店铺详情页所需字段（id、name、price、quantity、规格相关、finalPrice 等）。
- **cartTotal**：该店商品小计，由 `items` 计算得出，写入时可由调用方算好或由统一方法计算。
- **updatedAt**：便于过期清理或排序（可选）。

### 2.2 单条购物车商品（item）字段约定

与现有 `store-detail` / `checkout` 兼容，建议保留：

| 字段 | 类型 | 说明 |
|------|------|------|
| id | string | 商品 ID（与店铺商品列表一致） |
| name | string | 商品名 |
| price | number/string | 单价（或规格单价） |
| image | string | 图片 URL |
| quantity | number | 数量 |
| selectedSpec | object | 简单规格（旧逻辑兼容） |
| selectedSpecs | object | 规格组索引映射（与 specifications 对应） |
| finalPrice | string | 规格后的单价，用于展示与计算 |
| specifications | array | 可选，规格组结构（购物车展示规格文案时可从 store 拉取或存一份） |

同一商品不同规格视为不同 item（如现有规格弹窗逻辑：`id + selectedSpecs` 一致才合并数量）。

### 2.3 与结算页的对接（cartData）

结算页 `onLoad(options.cartData)` 期望的结构（与现有一致）：

```js
{
  cartItems: [...],   // 当前店 items
  cartTotal: 123,
  deliveryType: 'delivery',
  storeInfo: { _id, storeId, name, deliveryFee, minOrder, ... }
}
```

从全局购物车「某店桶」生成 `cartData` 时：  
`cartItems = bucket.items`，`cartTotal = bucket.cartTotal`（或由 items 实时算），`storeInfo = bucket.storeInfo`，`deliveryType = bucket.deliveryType`。

## 三、场景与流程

### 3.1 店铺详情内「加购」

- **触发**：点商品卡片「+」或规格弹窗内「加入购物车」。
- **逻辑**：
  1. 校验店铺状态：若 `businessStatus !== 'open'`，提示「店铺当前休息中」，不写入全局购物车。
  2. 按现有逻辑更新本页 `cartItems`、`products`、`displayProducts`、`cartTotal`（不变）。
  3. **同步到全局购物车**：  
     - 读取 `globalCart`，若无该 `storeId` 则新建桶，并写入 `storeInfo`（当前页的 `storeInfo`）、`deliveryType`。  
     - 将当前页的 `cartItems` 覆盖该店的 `items`，重算该店 `cartTotal`，写回 `globalCart`。  
  4. Toast「已添加到购物车」。
- **结果**：本页底部购物车栏与数量更新；全局购物车中该店数据与当前页一致。

### 3.2 店铺详情内「减数量 / 删除」

- **触发**：在购物车弹层或列表中点「-」或删除。
- **逻辑**：
  1. 按现有逻辑更新本页 `cartItems`、`cartTotal` 等。
  2. **同步到全局购物车**：用当前页该店的 `cartItems` 覆盖 `globalCart[storeId].items`，更新 `cartTotal`；若 `items.length === 0`，可从 `globalCart` 中删除该店键（或保留空 items，由购物车页按空处理）。
  3. 若清空，关闭弹层并更新底部栏。
- **结果**：本页与全局购物车保持一致。

### 3.3 店铺详情内「去结算」

- **触发**：底部「结算」或弹层内「去结算」。
- **逻辑**：**保持现有行为**：用当前页 `cartItems`、`cartTotal`、`storeInfo`、`deliveryType` 组装 `cartData`，`navigateTo` 结算页（URL 传参）。  
- **可选增强**：跳转前先执行一次「同步到全局购物车」（把当前页状态写回），再跳转，以便用户若从结算页返回再进购物车 Tab 时看到的是最新数据。

### 3.4 进入店铺详情时「拉取该店购物车」

- **触发**：`store-detail` 的 `onLoad` / `onShow`（若希望每次进店都看到最新全局数据，可在 `onShow` 做）。
- **逻辑**：
  1. 读取 `globalCart[storeId]`。
  2. 若有该店且 `items.length > 0`：  
     - 将 `items` 赋给本页 `cartItems`；  
     - 根据 `cartItems` 回填 `products` / `displayProducts` 中对应商品的 `quantity`（及规格），保证列表上的数量角标与购物车一致；  
     - 重算 `cartTotal` 并 setData。  
  3. 若无或空：本页 `cartItems` 置空，`cartTotal` 为 0。
- **结果**：用户从购物车 Tab 或从首页再次进入同一店铺时，看到的是已保存的购物车状态，而不是空白。

### 3.5 打开「购物车」Tab 页

- **触发**：底部 Tab 切到「购物车」。
- **逻辑**：
  1. `onShow` 时读取 `globalCart`。
  2. 将各店按 `updatedAt` 或首次加购时间排序（可选），渲染为「按店铺分组」的列表。
  3. 每个店铺块展示：店铺名、起送/配送费提示、该店 `items`（名称、规格、单价、数量、小计）、该店小计、是否达起送、按钮「去结算」。
  4. 若 `globalCart` 为空或没有任何一家有 item：展示空状态（图标 + 文案「购物车是空的」+「去逛逛」跳首页）。
- **结果**：用户看到所有店的加购情况，并可选择某一店去结算。

### 3.6 购物车页内「改数量 / 删除」

- **触发**：某商品后的「+」「-」或删除图标。
- **逻辑**：
  1. 更新内存中该店 `items`（数量加减或移除该项）。
  2. 重算该店 `cartTotal`。
  3. 写回 `globalCart`。
  4. setData 更新列表；若该店 items 为空，可从列表移除该店块（或展示「该店购物车为空」占位，并保留「去店铺看看」入口）。
- **结果**：全局购物车与购物车页展示一致；若用户再进该店铺详情，应拉取到已更新后的数据（见 3.4）。

### 3.7 购物车页内「去结算」（某店）

- **触发**：某店铺块上的「去结算」按钮。
- **校验**：
  - 若该店 `cartTotal < storeInfo.minOrder`：Toast「未达起送价 ¥xx」，不跳转。
  - 若该店 `storeInfo.businessStatus !== 'open'`：Toast「店铺当前休息中，暂不可结算」，不跳转（可选：从店铺接口拉一次最新状态再判断）。
- **逻辑**：
  1. 用该店桶组装 `cartData`（cartItems、cartTotal、storeInfo、deliveryType）。
  2. `wx.navigateTo({ url: '/subpackages/store/pages/checkout/index?cartData=' + encodeURIComponent(JSON.stringify(cartData)) })`。
- **结果**：进入与店铺详情内「去结算」相同的结算页，后续下单、支付流程一致。

### 3.8 从购物车页「去店铺」

- **触发**：某店铺块上的店铺名或「去店铺看看」。
- **逻辑**：`navigateTo` 店铺详情页，传入 `storeId`。店铺详情 onLoad/onShow 会从全局购物车拉取该店数据（3.4），展示一致。

### 3.9 结算页「返回」

- **触发**：结算页导航栏返回或左上角返回。
- **逻辑**：返回上一页（可能是店铺详情或购物车 Tab 页）。无需额外逻辑；若从购物车页进结算页，返回后购物车页 onShow 可再次读 `globalCart` 刷新（可选，因结算成功会跳转订单页，未提交则购物车未改动）。

## 四、入口与出口汇总

| 入口 | 出口 |
|------|------|
| 首页 / 其他 → 店铺详情 → 加购 | 店铺详情内「结算」→ 结算页 |
| 首页 / 其他 → 店铺详情 → 加购 | 切换 Tab 到「购物车」→ 购物车页选店「去结算」→ 结算页 |
| 底部 Tab「购物车」 | 空状态「去逛逛」→ 首页 |
| 底部 Tab「购物车」 | 某店「去结算」→ 结算页 |
| 底部 Tab「购物车」 | 某店「去店铺」→ 店铺详情 |
| 店铺详情 | 用户可主动切到购物车 Tab，再选店结算或编辑 |

## 五、反馈与边界

### 5.1 提示与引导

- 加购成功：Toast「已添加到购物车」。
- 删除/清空：Toast「已移除」或「已清空」；购物车页某店清空后可提示「该店已无商品」并保留「去店铺看看」。
- 未达起送：在购物车页该店块上显示「还差 ¥xx 起送」或「¥xx 起送」；点「去结算」时 Toast「未达起送价 ¥xx」。
- 店铺休息：店铺详情加购时拦截并提示；购物车页「去结算」时校验并提示「店铺当前休息中」。

### 5.2 空状态

- **购物车页无任何商品**：图标 +「购物车是空的」+「去逛逛」→ 首页。
- **某店块内无商品**：若实现上保留空桶，可显示「该店暂无商品」+「去店铺看看」；或直接不展示该店块。

### 5.3 异常与弱网

- 读写 `globalCart` 为本地同步，一般无网络依赖；若未来改为云购物车，需在「加载购物车」失败时展示重试或「使用本地缓存」的降级。
- 结算页仍依赖 URL 传参；若参数过长，可考虑先写临时 storage，结算页从 storage 读（需约定 key 与过期）。

### 5.4 多店铺一致性

- 在店铺详情修改数量或删除后，必须同步回 `globalCart`，这样从购物车 Tab 再进或从购物车页「去店铺」时，数据一致。
- 购物车页修改后写回 `globalCart`，再进店铺详情时通过 onShow 拉取，保证一致。

## 六、Tab 角标（可选）

- **购物车 Tab 角标**：显示「当前全局购物车中商品总件数」（各店 `items` 的 `quantity` 之和）。  
- 实现：在 `app.js` 或购物车页、店铺详情页，在每次更新 `globalCart` 后，调用 `wx.setTabBarBadge({ index: 2, text: String(count) })` 或 `wx.removeTabBarBadge({ index: 2 })`（index 为购物车在 tabBar.list 中的下标）。  
- 若总数为 0，可去掉角标。

## 七、与现有代码的对接要点

### 7.1 店铺详情页（store-detail）

- **加购/删减/清空**：在现有 setData 之后增加一步「同步到全局购物车」的封装调用（写入 `globalCart[storeId]`）。
- **onLoad/onShow**：增加「从 globalCart[storeId] 拉取 items 并回填 cartItems 与 products/displayProducts 数量」的逻辑。
- **去结算**：可保持现有 URL 传参；可选在跳转前先执行一次同步再跳转。

### 7.2 购物车 Tab 页（pages/cart/index）

- **onShow**：读取 `globalCart`，按店分组渲染；无数据时展示空状态。
- **列表**：每店一块，支持每项加减数量、删除；底部「去结算」「去店铺」。
- **去结算**：按 3.7 校验后组装 `cartData`，navigateTo 结算页。

### 7.3 结算页（checkout）

- **无需改入参**：仍通过 `options.cartData` 接收结构不变；从店铺详情或从购物车页进入均可。
- **可选**：若将来支持「从购物车页带临时 key」读取大 cartData，再扩展不迟。

### 7.4 工具方法建议

- **getGlobalCart()**：`wx.getStorageSync('globalCart')` 或 `{}`。
- **setStoreCart(storeId, payload)**：更新某店桶（storeInfo、items、cartTotal、deliveryType 等），写回 `globalCart`。
- **getStoreCart(storeId)**：返回 `globalCart[storeId]` 或 null。
- **removeStoreFromCart(storeId)**：删除某店键。
- **getCartTotalCount()**：遍历各店 items 的 quantity 之和，用于 Tab 角标。

可放在 `utils/cart.js`，供店铺详情与购物车页共用。

---

## 八、实施顺序建议

1. **utils/cart.js**：实现 globalCart 的读写、按店更新、总件数。
2. **store-detail**：onLoad/onShow 拉取该店购物车；加购/删减/清空后同步到 globalCart；去结算前可选同步。
3. **pages/cart/index**：onShow 读 globalCart 按店展示；列表内加减删；去结算/去店铺；空状态。
4. **Tab 角标**：在 cart 与 store-detail 更新 globalCart 后更新角标（可选）。

以上为购物车完整交互逻辑设计，可直接作为开发实现与联调依据。
